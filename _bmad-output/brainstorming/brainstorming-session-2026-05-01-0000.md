## Technique Execution Results

**What If Scenarios:**

- **Interactive Focus:** What if the app is not a tracker, but a personal museum of food life?
- **Key Breakthroughs:** Dining-out photos should reflect happiness, joy, and discovery. Home-cooking photos should reflect warmth, comfort, and a feeling of home.

**Captured Ideas:**

**[Emotional Design #1]**: Joy Outside, Home Inside
_Concept_: The app can intentionally separate the emotional language of dining out and home cooking. Dining-out memories should feel bright, social, exploratory, and happy; home-cooking memories should feel warm, intimate, grounded, and comforting.
_Novelty_: Instead of treating all food photos as the same content type, the product uses emotional context to shape layout, language, colors, and interactions.

**[Dining Out #2]**: Discovery as Happiness
_Concept_: Dining-out memories are happy in general, with discovery as the main emotional driver. Each restaurant entry can be framed as a place found, tried, and added to the user's personal food map.
_Novelty_: The restaurant section becomes a discovery journal rather than a review database; it prioritizes the joy of finding new places over ratings or critique.

**[Home Cooking #3]**: Comfort Made By Me
_Concept_: Home-cooking memories should feel like comfort because the user usually cooks foods they personally like, and because making the dish by hand makes it definitively home. The app should treat home cooking as self-made comfort, not just recipe storage.
_Novelty_: The cooking section centers authorship and emotional comfort instead of productivity, meal planning, or recipe management.

**[Product Language #4]**: Made and Found
_Concept_: The app can be organized around two simple verbs: Made for home-cooked comfort food, and Found for dining-out discoveries. This language is concise, emotionally clear, and flexible enough for both iOS navigation and web presentation.
_Novelty_: The app avoids generic labels like Cooking and Restaurants, using personal action words that reinforce the user's relationship with each memory.

**[Memory Design #5]**: Tiny Stories, Not Journals
_Concept_: Each food memory can include lightweight story fields without becoming a long-form diary. Made entries can capture why the user made the dish, comfort level, ingredients, and an optional note; Found entries can capture how the place was found, whether it was a first visit, restaurant/date details, and an optional note.
_Novelty_: The app adds emotional context while preserving low-friction photo capture and browsing.

**[Home Experience #6]**: Map for Found, Gallery for Made
_Concept_: The app can give each side a distinct browsing mode. Found memories naturally live on a map because discovery is place-based; Made memories naturally live in a cozy gallery because home cooking is personal, visual, and comfort-based.
_Novelty_: Rather than forcing one gallery pattern on every memory, the app lets the emotional meaning of each section determine its interface.

**[Platform #7]**: Full iOS and Full Web
_Concept_: The web app should support full create, edit, and browsing behavior like the iOS app, not merely a view-only companion. Both platforms should let the user manage Made and Found memories.
_Novelty_: The app becomes a durable personal archive usable wherever the user is, while still letting each platform emphasize its strengths.

**[Capture #8]**: One-Tap Memory Type
_Concept_: When adding a memory, the first choice is simply Made or Found. That choice changes the form fields, visual tone, and metadata.
_Novelty_: The app treats capture as emotional classification, not database entry.

**[Capture #9]**: Photo First, Details Second
_Concept_: The add flow starts with the photo, then asks for only the relevant details. For Made: dish and ingredients. For Found: restaurant and date.
_Novelty_: It matches how food memories happen: the photo is the anchor.

### Phase 3 Transition

The user requested completing SCAMPER idea development and moving toward implementation planning. Phase 3 refined the product through accepted substitutions, adaptation patterns, modifications, secondary uses, eliminations, and reversals.

## Phase 4: Solution Matrix

**MVP Platform Decision:** First version should be single-user/local-first.

**Rationale:** Local-first single-user scope reduces account, sync, and backend complexity, keeping the first version focused on the core Made / Found memory experience.

**Data Source Decision:** iOS and web should consume the same data source abstraction.

**Current Direction:** The data source is local for v1.

**Future Direction:** The same abstraction should allow moving to shared cloud storage later.

**Rationale:** The product should avoid divergent iOS and web data models. Even if v1 is local-first, the app should be designed around one shared memory schema and storage interface so future cloud sync is feasible.

**Photo Storage Decision:** Use local photo storage for v1, but design for cloud photo storage later.

**Rationale:** Photos should work locally in the first version while the memory schema and storage layer remain portable enough to support shared cloud storage in a future version.

**Timeline Decision:** Timeline should be a full v1 navigation tab.

**Final v1 Navigation:** Home / Made / Found / Timeline.

**Rationale:** Timeline is part of the core food-life archive experience, not a later add-on. It provides the combined chronological view of Made and Found memories while preserving their separate primary sections.

## Idea Organization and Prioritization

### Thematic Organization

**Theme 1: Made / Found Product Spine**

Made and Found are the heart of FoodLife. Made represents food the user cooked: comfort, home, ingredients, and self-made care. Found represents food the user discovered outside: restaurants, happiness, places, and dining memories.

Key ideas:

- Made and Found
- Joy Outside, Home Inside
- Comfort Made By Me
- Discovery as Happiness
- Two Emotional Worlds

**Pattern Insight:** The product becomes clear when every memory is either made by the user or found by the user.

**Theme 2: Photo-First Memory Capture**

Food photos are the primary artifact. Details support the photo rather than competing with it. Entry flows should be soft, lightweight, and type-specific.

Key ideas:

- Photo First, Details Second
- One-Tap Memory Type
- Tiny Stories, Not Journals
- Amplify Photos
- Soften Data Entry

**Pattern Insight:** The app should feel like saving a meaningful food memory, not filling out a database.

**Theme 3: Two Primary Browsing Worlds**

Made and Found should stay separate in their main sections because they carry different emotional meanings.

Key ideas:

- Map for Found, Gallery for Made
- Make Made More Gallery-First
- Personal Food Map
- Photo Pin Preview
- Detail Card Mood Shift

**Pattern Insight:** Made should default to a warm gallery. Found should default to a discovery map.

**Theme 4: Seasonal Reflection and Timeline**

The long-term payoff is seeing food life reflected over time, especially through seasonal chapters.

Key ideas:

- Seasonal Landing
- Food Life Chapter
- Gentle Recap, Not Analytics
- Seasonal Cover Photo
- Food Life Timeline
- This Time Last Year

**Pattern Insight:** Home should be a seasonal reflection, not a dashboard. Timeline should be a full v1 navigation tab.

**Theme 5: Local-First, Cloud-Ready Architecture**

The first version should reduce infrastructure burden while preserving future portability.

Key ideas:

- Single-user/local-first v1
- Same data source abstraction for iOS and web
- Local photo storage for v1
- Cloud-ready storage design
- Full iOS and full web support

**Pattern Insight:** v1 should prove the memory experience locally while keeping the data model ready for future shared cloud storage.

### Prioritization Results

**Top Priority Ideas**

1. **Made / Found as the core structure**  
   This is the strongest product decision. It gives the app emotional clarity, navigation clarity, and data-model clarity.

2. **Seasonal Home as reflection**  
   This is the strongest long-term delight mechanism. It makes the app feel alive after months or years of use.

3. **Made gallery + Found map**  
   This gives each section the right browsing metaphor: comfort at home, discovery outside.

4. **Photo-first, soft add flow**  
   The app must stay lightweight. Capture should begin with Made or Found, then photo, then only relevant details.

5. **Local-first shared data model**  
   This keeps MVP buildable while avoiding a dead-end architecture.
